## Summary

Phase 1 is a pure backend phase. It establishes the runnable Spring Boot skeleton, correct JWT authentication with Spring Security 6, seeded users via Flyway, and the local development environment (Docker Compose + Makefile). No Angular frontend code is created in this phase — frontend work begins in Phase 5.

The most important correctness concern is the JWT filter registration. Spring Security 6 and the project decision both mandate that `JwtAuthFilter` is NOT annotated with `@Component`. The filter must be instantiated as a `@Bean` in a configuration class (without `@Component` on the filter class itself), then injected into `SecurityFilterChain`. This avoids double-registration in the servlet container. A secondary correctness concern is the `SecurityContextHolderStrategy` pattern: when setting authentication in the filter, use `SecurityContextHolder.getContextHolderStrategy().getContext().setAuthentication(auth)` to guarantee compatibility with Spring Security 6's holder strategy mechanism.

CORS must be configured entirely within `SecurityFilterChain` via a `CorsConfigurationSource` bean. Any `WebMvcConfigurer`-based CORS configuration is bypassed by the security filter layer for secured endpoints. Seed users are created via a Flyway migration (V2__seed_users.sql) using pre-computed BCrypt hashes — not via a `DataInitializer` bean (which would require checking existence and is fragile at startup).

**Primary recommendation:** Build the backend in this sequence: (1) Maven pom.xml, (2) Flyway migrations + Docker Compose, (3) User entity + repository, (4) JwtService + JwtAuthFilter + SecurityConfig, (5) AuthController, (6) Integration tests, (7) Makefile.

### Pattern 1: SecurityFilterChain with JwtAuthFilter as @Bean

`JwtAuthFilter` is declared as a `@Bean` in a separate `@Configuration` class (or inside `SecurityConfig` with a factory method). The filter class has NO `@Component` annotation. This is the critical difference from incorrect examples in STACK.md and ARCHITECTURE.md (which show `@Component` — those documents contain an error that CONTEXT.md locked decisions override).

```java
// Source: CONTEXT.md locked decision + PITFALLS.md
// security/JwtAuthFilter.java — NO @Component annotation
@RequiredArgsConstructor
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        final String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        final String token = authHeader.substring(7);
        final String username = jwtService.extractUsername(token);

        if (username != null
                && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);
            if (jwtService.isTokenValid(token, userDetails)) {
                var authToken = new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());
                authToken.setDetails(
                        new WebAuthenticationDetailsSource().buildDetails(request));
                // Spring Security 6: use strategy-aware setter
                SecurityContextHolder.getContextHolderStrategy()
                        .getContext().setAuthentication(authToken);
            }
        }
        filterChain.doFilter(request, response);
    }
}
```

```java
// config/SecurityConfig.java
@Configuration
@EnableWebSecurity
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Bean
    public JwtAuthFilter jwtAuthFilter() {
        // Constructed here — never picked up by Spring as a servlet filter
        return new JwtAuthFilter(jwtService, userDetailsService);
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            .sessionManagement(s ->
                s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(
                    "/api/auth/**",
                    "/v3/api-docs/**",
                    "/swagger-ui/**",
                    "/swagger-ui.html",
                    "/actuator/health"
                ).permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter(), UsernamePasswordAuthenticationFilter.class)
            .build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration config)
            throws Exception {
        return config.getAuthenticationManager();
    }

    @Bean
    CorsConfigurationSource corsConfigurationSource(
            @Value("${cors.allowed-origins:http://localhost:4200}") String allowedOrigins) {
        var config = new CorsConfiguration();
        config.setAllowedOriginPatterns(List.of(allowedOrigins.split(",")));
        config.setAllowedMethods(List.of("GET","POST","PUT","PATCH","DELETE","OPTIONS"));
        config.setAllowedHeaders(List.of("*"));
        config.setAllowCredentials(true);
        var source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/api/**", config);
        return source;
    }
}
```

### Pattern 2: JJWT 0.12.6 API

```java
// Source: STACK.md (verified against Maven Central jjwt-api 0.12.6)
// security/JwtService.java
@Service
@Slf4j
public class JwtService {

    @Value("${jwt.secret}")
    private String jwtSecret;

    @Value("${jwt.expiration-ms:3600000}")
    private long expirationMs;

    private SecretKey signingKey() {
        return Keys.hmacShaKeyFor(Decoders.BASE64.decode(jwtSecret));
    }

    public String generateToken(UserDetails userDetails) {
        return Jwts.builder()
            .subject(userDetails.getUsername())
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expirationMs))
            .signWith(signingKey())
            .compact();
    }

    public String extractUsername(String token) {
        return parseClaims(token).getSubject();
    }

    public boolean isTokenValid(String token, UserDetails userDetails) {
        final String username = extractUsername(token);
        return username.equals(userDetails.getUsername()) && !isTokenExpired(token);
    }

    private Claims parseClaims(String token) {
        return Jwts.parser()
            .verifyWith(signingKey())
            .build()
            .parseSignedClaims(token)   // 0.12.x API — NOT parseClaimsJws()
            .getPayload();
    }

    private boolean isTokenExpired(String token) {
        return parseClaims(token).getExpiration().before(new Date());
    }
}
```

### Anti-Patterns to Avoid

- **@Component on JwtAuthFilter:** Causes the filter to run twice — once in the servlet container's default filter chain, once inside Spring Security. The filter must be instantiated as a `@Bean` (or inline in `securityFilterChain`) but the class must have no `@Component` / `@Service` annotation.
- **CORS in WebMvcConfigurer only:** Spring Security's filter layer runs before the MVC dispatcher. `WebMvcConfigurer.addCorsMappings()` never fires for requests that Spring Security intercepts. All CORS config must go inside `SecurityFilterChain`.
- **allowedOrigins("*") with allowCredentials(true):** This combination is rejected at startup by Spring Security. Use `allowedOriginPatterns()` instead of `allowedOrigins()` when credentials are required.
- **antMatchers / authorizeRequests:** Removed in Spring Security 6. Causes compilation failure immediately. Use `requestMatchers()` and `authorizeHttpRequests()`.
- **Editing applied Flyway migrations:** Breaks all environments with checksum mismatch. V1, V2 files are immutable once committed.
- **DataInitializer bean for seeding:** Fragile at startup, not idempotent without existence checks, not visible in migration history. Use V2__seed_users.sql instead.
- **SecurityContextHolder.getContext().setAuthentication():** In Spring Security 6, use the strategy-aware form: `SecurityContextHolder.getContextHolderStrategy().getContext().setAuthentication()`. Using the direct form can cause 403 when a non-default holder strategy is active.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Token signing/parsing | Custom HMAC JWT code | JJWT 0.12.6 | Edge cases: clock skew, algorithm confusion attack, expiry handling |
| Password hashing | Custom hash function | BCryptPasswordEncoder | BCrypt handles salting, cost factor, timing-safe comparison |
| CORS preflight handling | Custom OPTIONS handler | CorsConfigurationSource bean | Spring Security handles OPTIONS before routing; custom handlers miss this |
| .env loading | Custom PropertySource | spring-dotenv 4.0.0 | Participates in Spring's relaxed property resolution natively |
| DB schema management | ddl-auto=create | Flyway migrations | Flyway gives reproducible, version-tracked, environment-safe schema |

**Key insight:** Spring Security 6 provides all the primitives needed for JWT auth. The only custom code is the filter (which extracts the token and calls `JwtService`) and the service itself. Everything else — BCrypt, CORS, session management, method security — is configuration.

## Common Pitfalls

### Pitfall 1: JWT Filter @Component Double-Registration

**What goes wrong:** Filter runs twice per request. On the second pass the security context is already populated (or cleared, depending on order), causing either 403 or authentication leakage.

**Why it happens:** Spring Boot registers all `@Component` beans that extend `Filter` or `OncePerRequestFilter` automatically in the servlet container's default filter chain. When also added to `SecurityFilterChain` via `addFilterBefore`, it runs twice.

**How to avoid:** No `@Component` annotation on `JwtAuthFilter`. Create it via `@Bean` factory method in `SecurityConfig` or another `@Configuration` class.

**Warning signs:** Auth events logged twice in structured logs; debug logging shows filter invocation twice per request.

---

### Pitfall 2: SecurityContextHolderStrategy Race Condition

**What goes wrong:** Valid JWT is parsed and principal set, but subsequent filters see `AnonymousAuthenticationToken` and return 403.

**Why it happens:** Spring Security 6 introduced a pluggable `SecurityContextHolderStrategy`. If custom code writes to `SecurityContextHolder.getContext()` directly while a different strategy is active, the downstream `AuthorizationFilter` reads from the strategy's context and sees nothing.

**How to avoid:** Use `SecurityContextHolder.getContextHolderStrategy().getContext().setAuthentication(auth)` in `JwtAuthFilter.doFilterInternal()`.

**Warning signs:** Filter logs show token validated successfully, but controller receives 403 or ANONYMOUS principal.

---

### Pitfall 3: CORS Preflight Returns 401

**What goes wrong:** Angular dev server makes CORS preflight (`OPTIONS`) requests; they return 401 because Spring Security intercepts them without an `Authorization` header.

**Why it happens:** `WebMvcConfigurer.addCorsMappings()` is processed by the DispatcherServlet — AFTER Spring Security runs. Spring Security sees an unauthenticated OPTIONS request and rejects it.

**How to avoid:** Configure CORS exclusively in `SecurityFilterChain` using a `CorsConfigurationSource` bean passed to `.cors(cors -> cors.configurationSource(...))`. Spring Security handles OPTIONS before checking authentication.

**Warning signs:** Preflight OPTIONS returns 401/403; GET/POST work from Swagger UI (same origin) but fail from Angular (cross-origin).

---

### Pitfall 4: Flyway Double-Underscore Naming

**What goes wrong:** Migration file named `V1_create_users.sql` (one underscore) is silently ignored. Schema is never created. Application starts, Flyway reports no migrations applied.

**Why it happens:** Flyway's filename parser requires the separator to be exactly two underscores `__`. One underscore is a valid filename character and Flyway treats the whole suffix as the description name, failing to parse the version.

**How to avoid:** Always use double underscore: `V1__create_users.sql`, `V2__seed_users.sql`. Validate by checking that Flyway logs "Successfully applied N migrations" on first startup.

**Warning signs:** Flyway reports 0 migrations found; application starts but cannot authenticate because users table doesn't exist.

---

### Pitfall 5: Docker Compose `localhost` vs Service Name

**What goes wrong:** Backend container connects to `localhost:5432` and gets `Connection refused` because Postgres is in a different container.

**Why it happens:** Inside Docker Compose, each container has its own network namespace. `localhost` refers to the container itself.

**How to avoid:** Use the Compose service name (`db`) in datasource URL: `jdbc:postgresql://db:5432/opspilot`. Also use `depends_on: db: condition: service_healthy` so Spring Boot does not start before Postgres is ready.

**Warning signs:** Spring Boot container exits immediately with `Connection refused` on port 5432; `docker compose logs backend` shows HikariCP pool timeout.

---

### Pitfall 6: Java Version Mismatch

**What goes wrong:** `pom.xml` sets `<java.version>21</java.version>` but system has Java 25 installed. Maven reports warning; if set incorrectly Spring features may not activate.

**Why it happens:** `spring-boot-starter-parent` 3.3.x defaults to Java 17. Without overriding `<java.version>`, virtual threads (`spring.threads.virtual.enabled=true`) may not activate.

**How to avoid:** Set `<java.version>21</java.version>` in `<properties>`. Java 25 JDK compiles to Java 21 bytecode with `--release 21`. All Java 21 APIs (virtual threads, records, switch expressions) are available.

**Warning signs:** `spring.threads.virtual.enabled=true` has no effect; Tomcat thread pool is not virtualised.

### AuthService

```java
// Source: ARCHITECTURE.md + STACK.md
@Service
@RequiredArgsConstructor
@Slf4j
public class AuthService {

    private final AuthenticationManager authManager;
    private final JwtService jwtService;
    private final UserRepository userRepository;

    public AuthResponse login(LoginRequest request) {
        try {
            var authentication = authManager.authenticate(
                new UsernamePasswordAuthenticationToken(request.email(), request.password()));
            var userDetails = (UserDetails) authentication.getPrincipal();
            var token = jwtService.generateToken(userDetails);
            var user = userRepository.findByEmail(request.email())
                .orElseThrow();
            log.info("AUTH login success: email={} role={}", user.getEmail(), user.getRole());
            return new AuthResponse(token, user.getEmail(), user.getRole().name());
        } catch (BadCredentialsException ex) {
            log.warn("AUTH login failure: email={}", request.email());
            throw ex;
        }
    }
}
```

| # | Claim | Section | Risk if Wrong |

**Note on A1:** The Python `bcrypt` library generates `$2b$` hashes (RFC-compliant). This is consistent with the jBCrypt source code behaviour [ASSUMED — not verified against Spring Security 6 source in this session]. If A1 is wrong, replace seed hashes by running `new BCryptPasswordEncoder(12).encode("admin123")` in a test and using the `$2a$` output.

### Known Threat Patterns for JWT + Spring Boot

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| JWT algorithm confusion (none/alg=none) | Tampering | JJWT 0.12.x rejects unsigned tokens; verifyWith(signingKey()) enforces HMAC |
| Weak JWT secret | Information Disclosure | Minimum 32 bytes for HS256; recommend 64 bytes; from env var only |

| SQL injection on email | Tampering | Spring Data JPA parameterised queries; `findByEmail(String email)` |
| XSS token theft | Elevation of Privilege | JWT stored in sessionStorage (STATE.md decision); CSP header recommended |
